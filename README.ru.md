# Agent Reflection

[English](README.md) · **Русский** · [Español](README.es.md)

**Локальный аудит сессий и рекомендации по workflow агента для [Claude Code](https://claude.com/claude-code).**

Agent Reflection наблюдает за вашими сессиями с кодовым агентом через hooks
Claude Code, хранит **безопасную для приватности локальную телеметрию**,
выявляет неэффективные workflow агента и после каждой сессии выдаёт
рекомендации, основанные на фактических данных.

```text
[███░░░░░░░] 32% 65K · ↻ 521K
```

## Зачем

Сессия закончилась, контекст разросся до 80%, и вы понятия не имеете, что его
съело. Двадцать чтений файлов до первой правки? Тест, упавший шесть раз подряд?
Компакция где-то посередине?

И счёт растёт не линейно: каждый запрос заново отправляет весь диалог, поэтому
за мусорный дамп файла в начале вы платите в каждом последующем ходе. Стоимость
растёт как *квадрат* длины сессии — шум, убранный рано, стоит куда больше, чем
шум, убранный поздно.

Agent Reflection — это **петля обратной связи для разработчика, а не для
модели**: короткий, основанный на фактах разбор вашей собственной сессии — куда
ушли токены, какие части были дорогими, какие из них мог бы взять на себя
дешёвый subagent с доступом только на чтение — написанный локально, при этом
ни промпты, ни код не покидают машину.

## Что это — и чем это не является

Agent Reflection помогает снизить **суммарную стоимость успешно сделанной
работы**, а не просто количество токенов. Он поощряет простой workflow:

1. Использовать дешёвую модель для ограниченного исследования только на чтение.
2. Использовать стандартную модель для чётко очерченной реализации.
3. Эскалировать к более сильной рассуждающей модели только тогда, когда это
   оправдано фактами.
4. Держать шумные операции вне основного окна контекста.

> [!IMPORTANT]
> **Это не гарантированный оптимизатор моделей.** Agent Reflection никогда не
> утверждает, что знает универсально оптимальную модель, и никогда не
> утверждает, что другая модель *дала бы* более дешёвый или лучший результат.
> Он лишь показывает **наблюдаемые сигналы** — исследование только на чтение в
> основном контексте, повторяющиеся циклы неудач, давление на контекст и
> ограниченные сегменты исследования — и подаёт каждую рекомендацию как
> кандидата, который оцениваете вы.

После установки всё работает **полностью офлайн**. Agent Reflection не делает
сетевых запросов и никуда не отправляет код, промпты, вывод инструментов,
телеметрию или метаданные репозитория.

## Установка

**Требования:** Node.js 22+, pnpm, Claude Code.

Репозиторий одновременно является маркетплейсом Claude Code
(`.claude-plugin/marketplace.json`), поэтому устанавливается прямо с GitHub —
никакого реестра. `dist/` закоммичен, так что собирать ничего не нужно.

```bash
claude plugin marketplace add mikhkonkov/agent-reflection
claude plugin install agent-reflection@agent-reflection --scope user

# runtime dependencies for the installed copy (better-sqlite3 is native).
# Hooks run from the version-scoped plugin cache, not from the marketplace clone:
pnpm install --prod --dir \
  "$(ls -d ~/.claude/plugins/cache/agent-reflection/agent-reflection/*/ | tail -1)"
```

Без этого шага hooks не найдут `better-sqlite3`, ничего не запишут и молча
завершатся с кодом `0` — поэтому повторяйте команду после каждого обновления
плагина, которое ставит его в новый каталог версии.

После этого перезапустите Claude Code, чтобы hooks загрузились.

Чтобы CLI `agent-reflection` появился в вашем `PATH`, сделайте symlink из клона
маркетплейса. Этот клон — **вторая копия**, отдельная от кеша плагина выше,
поэтому ему нужны собственные зависимости:

```bash
pnpm install --prod --dir ~/.claude/plugins/marketplaces/agent-reflection

ln -sf ~/.claude/plugins/marketplaces/agent-reflection/dist/cli/index.js \
  ~/.local/bin/agent-reflection
```

Пропустите `pnpm install` — и CLI упадёт на `Cannot find package 'commander'`,
хотя hooks продолжат работать: эти две копии ломаются независимо друг от друга.

Подойдёт любой каталог из вашего `PATH`; `~/.local/bin` — просто
распространённый вариант по умолчанию.

Затем в каждом репозитории, за которым хотите наблюдать:

```bash
agent-reflection init
```

Это необязательно — хранилище создаётся автоматически при первой сессии в
репозитории. `init` лишь заранее пишет файл конфигурации и предлагает
установить statusline (`--statusline` / `--skip-statusline`, чтобы ответить
неинтерактивно).

Проверить можно командой `claude plugin list`.

<details>
<summary>Если вы запускаете Claude Code со своим <code>CLAUDE_CONFIG_DIR</code></summary>

Плагины устанавливаются отдельно для каждого каталога конфигурации. У
отдельного профиля — скажем, у shell-алиаса, задающего
`CLAUDE_CONFIG_DIR=~/.claude-work` — свой список плагинов, и установка в
`~/.claude` для него невидима: ни skill, ни hooks не появятся.

Экспортируйте ту же переменную и выполните установку со скорректированными
путями:

```bash
export CLAUDE_CONFIG_DIR=~/.claude-work

claude plugin marketplace add mikhkonkov/agent-reflection
claude plugin install agent-reflection@agent-reflection --scope user
pnpm install --prod --dir \
  "$(ls -d "$CLAUDE_CONFIG_DIR"/plugins/cache/agent-reflection/agent-reflection/*/ | tail -1)"
```

Symlink в `PATH` подчиняется тому же правилу — клон маркетплейса тоже лежит
внутри этого каталога конфигурации:

```bash
pnpm install --prod --dir "$CLAUDE_CONFIG_DIR"/plugins/marketplaces/agent-reflection

ln -sf "$CLAUDE_CONFIG_DIR"/plugins/marketplaces/agent-reflection/dist/cli/index.js \
  ~/.local/bin/agent-reflection
```

Одного symlink достаточно, сколько бы каталогов конфигурации у вас ни было:
любой клон отдаёт один и тот же CLI, лишь бы в этом клоне были установлены
зависимости.

Повторите для каждого используемого каталога конфигурации. Если
`claude plugin list` сообщает `✘ failed to load … not found in marketplace`,
значит в этом каталоге конфигурации осталась устаревшая запись: удалите плагин
и его маркетплейс, затем установите заново.

</details>

> [!NOTE]
> `claude plugin marketplace update agent-reflection` подтягивает новую версию;
> всегда выполняйте после этого строку `pnpm install --prod` — обновление
> попадает в новый пустой каталог версии внутри кеша плагина.

> [!NOTE]
> `better-sqlite3` — нативный модуль. Если для вашей версии Node нет готового
> бинарника, pnpm соберёт его из исходников (build-скрипты разрешены в
> `package.json`). При ошибке "Could not locate the bindings file" выполните
> `pnpm rebuild better-sqlite3`.

### Удаление

```bash
claude plugin uninstall agent-reflection@agent-reflection --scope user --yes
claude plugin marketplace remove agent-reflection
```

Удаление плагина полностью прекращает сбор данных.

### Hooks

Hooks регистрируются на `SessionStart`, `UserPromptSubmit`, `PreToolUse`,
`PostToolUse`, `SubagentStop`, `PreCompact` и `SessionEnd`. Каждое событие
запускает `hooks/hook-router.mjs`, который читает JSON хука со stdin,
нормализует его в безопасное для приватности событие и сохраняет локально.
Роутер **всегда завершается с кодом `0`**, поэтому проблема с телеметрией
никогда не сможет заблокировать Claude Code или вызов инструмента.

## Использование CLI

```bash
agent-reflection init                              # create local storage + config
agent-reflection report latest                     # print the most recent session report
agent-reflection report current                    # the session in progress right now
agent-reflection report previous                   # the last session that finished
agent-reflection report <session-id>               # print a specific report
agent-reflection sessions                          # list recent sessions
agent-reflection sessions --repo .                 # scope to the current repository
agent-reflection stats --days 30                   # aggregate stats (no cost estimates)
agent-reflection config show
agent-reflection config set privacy.storeRawPayloads false
```

## Как работают отчёты

На `SessionEnd` Agent Reflection:

1. Загружает все события сессии.
2. Строит детерминированные агрегированные метрики.
3. Выполняет пять локальных проверок на основе правил.
4. Сохраняет рекомендации и пишет Markdown-отчёт в
   `.agent-reflection/reports/<YYYY-MM-DD>-<session-id>.md`.
5. При необходимости печатает однострочную сводку.

| Правило | Сигнал |
|---|---|
| `excessive-main-context-exploration` | Много исследования только на чтение в основном контексте |
| `repeated-execution-failure` | Цикл «правка → неудачный запуск» |
| `model-escalation-candidate` | Непродуктивный цикл, который может требовать независимой диагностики |
| `context-pressure` | Компакция и/или большой наблюдаемый вывод |
| `cheap-subagent-candidate` | Ограниченный сегмент исследования только на чтение |

Отчёты **детерминированы**: одинаковые входные события дают побайтово
идентичное тело отчёта.

## Встроенные агенты

Три subagent'а кодируют маршрутизацию, которую поощряет этот инструмент:

| Агент | Модель | Назначение |
|---|---|---|
| `explore-cheap` | Haiku, только чтение | Поиск файлов, трассировка путей выполнения, описание модулей, сбор фактов. Без правок, без Bash. |
| `implement-standard` | Sonnet | Чётко очерченная реализация с узкой валидацией и минимальными изменениями. |
| `architect-escalation` | Opus, только чтение | Независимая диагностика повторяющихся неудач, неоднозначной архитектуры, миграций, конкурентности или сложной отладки *до* дальнейших правок. |

Agent Reflection никогда не запускает subagent и не меняет конфигурацию сам по
себе; skills всегда сначала спрашивают.

## Чтение отчёта прямо из Claude Code

Чтобы увидеть результат, не нужно выходить из сессии. Запустите встроенный
skill в её конце:

```text
/agent-reflection:agent-reflection-report
```

Он загружает последний отчёт и разбирает главные рекомендации вместе с их
обоснованием — то же содержимое, что и у `agent-reflection report latest`, но
прямо в контексте, где можно задать уточняющие вопросы по находке.

## Индикатор контекста в statusline

Отчёт говорит, куда ушли токены, *после* сессии; statusline говорит об этом
**пока всё ещё происходит**. Иначе рост контекста незаметен — ничто не
предупреждает вас, пока Claude Code не выполнит компакцию, не выбросит первую
половину диалога, и агент не начнёт перечитывать файлы, которые уже знал:

```text
[███░░░░░░░] 32% 65K · ↻ 521K
```

Полоса заполнения, доля занятого окна контекста, токены в нём прямо сейчас и
`↻` — суммарный объём за сессию, включая всё уже скомпактированное.

Зелёный ниже 60%, жёлтый до 85%, красный выше — это диапазон, где компакция
становится вероятной. Жёлтый — сигнал действовать, пока запас ещё есть:
завершить текущую нить, отдать следующее исследование дешёвому subagent'у
только на чтение или начать новую сессию, вместо того чтобы позволить компакции
решать, что будет забыто.

Пока работают subagent'ы, каждая строка в панели агентов получает свой
индикатор, привязанный к контексту этого агента, а не к основному:

```text
explore-cheap        [██░░░░░░░░] 18% 36K · locate auth middleware
architect-escalation [█████████░] 85% 170K · diagnose repeated failure
```

**Строки subagent'ов ставятся сами** вместе с плагином (`settings.json` →
`subagentStatusLine`). **Основной statusline плагин зарегистрировать не может**
в Claude Code — он живёт в вашем собственном файле настроек, — поэтому плагин
упоминает о нём **один раз**, при первой сессии в репозитории, и настраивает
его только с вашего согласия:

```bash
agent-reflection init                    # asks before touching your settings
agent-reflection init --statusline       # install it, no prompt (scripts, CI)
agent-reflection init --skip-statusline  # leave the statusline alone

agent-reflection config set statusline.promptOnSessionStart false   # never ask
```

Существующий `statusLine` не затирается: он переносится в
`AGENT_REFLECTION_STATUSLINE_CHAIN`, отображается как префикс и
восстанавливается при удалении. Скрипты копируются в
`~/.claude/agent-reflection/statusline/`, и `statusLine` указывает на эту копию,
поэтому перемещение или удаление рабочей копии репозитория не ломает индикатор.

Это чистые bash + awk (`statusline/`), потому что statusline перерисовывается
постоянно, а старт node — это заметная задержка. Показания берутся из полей
`context_window`, которые Claude Code передаёт на stdin — те же числа, что и у
`/context`, — с откатом на разбор транскрипта в старых версиях.

## Гарантии приватности

По умолчанию Agent Reflection **никогда не сохраняет**: полный текст промпта,
полный ввод инструмента, полный вывод инструмента, исходный код, вывод
терминала/тестов, значения переменных окружения, секреты, API-ключи, полные
абсолютные пути или внешние URL.

Он хранит только агрегированную, нечувствительную телеметрию: id сессии,
хешированный идентификатор репозитория, имя каталога репозитория, имя ветки,
имя события, временные метки, имя и классификацию инструмента, флаги успеха,
**длины в символах** ввода/вывода, длительности, относительные пути (только
когда их безопасно вывести из `cwd`), количество путей, нормализованные
категории ошибок, жизненный цикл subagent'ов, триггеры компакции и **хеш**
SHA-256 каждого промпта (но не его текст).

Хранение сырых полезных нагрузок существует лишь как явно выключенная по
умолчанию опция (`privacy.storeRawPayloads`, `privacy.storePromptText`,
`privacy.storeToolOutput` — все `false`). Но даже при включении вероятные
секреты (API-ключи, bearer-токены, приватные ключи, пароли, присваивания
переменных окружения вида `KEY=value`) вычищаются до того, как что-либо будет
записано.

### Структура хранилища

Внутри репозитория, когда он доступен для записи:

```text
<repository-root>/.agent-reflection/
├── agent-reflection.db        # SQLite database
├── config.json
├── events/<session-id>.jsonl
└── reports/<YYYY-MM-DD>-<session-id>.md
```

Иначе — запасное хранилище на каждый репозиторий с ключом из стабильного
SHA-256 от абсолютного пути к репозиторию (сам путь никогда не попадает в имя
каталога):

```text
~/.agent-reflection/projects/<repository-hash>/
```

## Удаление всех локальных данных

Все данные локальны. Чтобы удалить их:

```bash
rm -rf .agent-reflection                        # repository-local storage
rm -rf ~/.agent-reflection/projects/<hash>      # fallback storage for a repo
rm -rf ~/.agent-reflection                      # every project's fallback storage
```

## Ограничения

- **Рекомендации — это эвристики, а не приговоры.** Они отмечают наблюдаемые
  паттерны; они не доказывают, что другая модель или маршрут были бы лучше.
- **Пересчёт токенов в деньги намеренно не делается.** Размеры измеряются в
  символах/байтах наблюдаемого ввода/вывода, а не в оплаченных токенах.
- **Отображение хуков.** Реальный Claude Code не предоставляет ни
  `PostToolUseFailure`, ни хука старта subagent'а. Agent Reflection выводит
  *неудачу* инструмента из полезной нагрузки `PostToolUse` и трактует вызов
  `Task` в `PreToolUse` как сигнал старта subagent'а. Это производные сигналы,
  и они могут отличаться от релиза к релизу Claude Code.
- Оценка «ходов» — задокументированная эвристика (`prompts + main-agent tool
  calls`), а не точный подсчёт ходов диалога.
- **Нет петли обратной связи по результату.** Каждое правило — эвристика,
  оценённая только по форме сессии; нигде не фиксируется, была ли работа
  действительно успешной, поэтому шумная, но успешная сессия выглядит так же,
  как застрявшая. Ручные метки результата (`accepted` / `rework` / `failed`)
  были выпущены и убраны — никто не занимается рутинной разметкой каждой
  сессии, так что данных не оказывалось тогда, когда они требовались правилам.
  Стоит вернуться к этому как к **выводимому** результату: коммит после
  сессии, следующая сессия, открывающая те же файлы, неудачи, сгруппированные
  в конце. Неиспользуемая колонка `sessions.user_outcome` сохранена именно для
  этого.
- Поддерживается только Claude Code — не Cursor, не Codex, не Windsurf и не
  другие CLI.
- Ни облака, ни дашборда, ни аутентификации, ни автоматического переключения
  моделей, ни автоматических изменений кода или запуска subagent'ов.
